# Design: Safari 12 WASM Compatibility via JS Transpilation

**Date:** 2026-03-29
**Status:** Approved

## Problem

The tablet UI (`/ui/tablet`) shows a blank page on an older iPad running iOS 12.5.8 (Safari 12). The iPad supports WebAssembly, but the JavaScript glue code generated by Trunk/wasm-bindgen uses ES2022 syntax that Safari 12 cannot parse:

- **Top-level `await`** in the inline `<script>` in `index.html` (ES2022, requires Safari 15+)
- Potential future use of `?.` (optional chaining, ES2020) or `??` (nullish coalescing, ES2020) by wasm-bindgen updates

Safari 12 supports up to ES2017. When it encounters unsupported syntax, the entire module fails to parse silently — blank page, no error visible to the user.

WASM itself works fine in Safari 12. Only the JS bootstrap is the problem.

## Solution

Add a post-build transpilation step using **esbuild** that rewrites the Trunk-generated JS to ES2017 (Safari 12 compatible). The WASM binary is untouched.

One transpiled bundle serves both operator and tablet. Modern browsers execute transpiled ES2017 JS identically — no performance or feature difference.

## Changes

### 1. New build script: `scripts/build-ui.sh`

Shared script called by all CI workflows and local dev:

```bash
#!/usr/bin/env bash
set -euo pipefail

cd crates/presenter-ui

# Step 1: Normal Trunk build
trunk build --release

# Step 2: Transpile JS glue to Safari 12 compatible syntax
esbuild dist/presenter-ui-*.js \
  --target=safari12 \
  --outdir=dist/ \
  --allow-overwrite

# Step 3: Fix top-level await in index.html inline script
# Trunk generates: const wasm = await init({...}); ... dispatchEvent(...)
# Wrap in async IIFE: (async () => { ... })();
sed -i 's|<script type="module">|<script type="module">(async () => {|' dist/index.html
sed -i 's|</script>|})();</script>|' dist/index.html
```

### 2. CI workflow changes

Three workflow files call the shared script instead of `trunk build --release` directly:

- **`pipeline.yml`** (line ~399): Replace `trunk build --release` with `bash ../../scripts/build-ui.sh`
- **`deploy.yml`** (line ~48): Same replacement
- **`release.yml`** (line ~46): Same replacement

Each workflow also needs esbuild installed. On GitHub-hosted runners (ubuntu-latest), add:

```yaml
- name: Install esbuild
  run: npm install -g esbuild
```

### 3. Files touched

| File | Change |
|------|--------|
| `scripts/build-ui.sh` | **New** — shared build + transpile script |
| `.github/workflows/pipeline.yml` | Install esbuild, call build-ui.sh |
| `.github/workflows/deploy.yml` | Install esbuild, call build-ui.sh |
| `.github/workflows/release.yml` | Install esbuild, call build-ui.sh |

### 4. What does NOT change

- WASM binary — untouched, not processed by esbuild
- Leptos components — no Rust code changes
- Tablet/operator UI behavior — identical on all browsers
- `index.html` template (`crates/presenter-ui/index.html`) — unchanged, the fixup is on the generated `dist/index.html`
- Asset serving code in `presenter-server` — unchanged
- Build time impact — ~100ms added for esbuild transpilation

## Verification Plan

1. **Local build**: Run `scripts/build-ui.sh`, inspect `dist/presenter-ui-*.js` for ES2022+ syntax (should find none)
2. **Local build**: Inspect `dist/index.html` — confirm no top-level `await` (wrapped in async IIFE)
3. **iPad test**: Open `http://10.77.9.205/ui/tablet` on the iPad (iOS 12.5.8) — page loads and is interactive
4. **E2E tests**: Existing Playwright tests pass (modern browsers handle transpiled JS identically)
5. **Operator test**: `/ui/operator` works normally on desktop browser

## Risk Assessment

- **Low risk**: esbuild is widely used (Vite, Snowpack), transpilation is a well-understood operation
- **Future-proof**: When wasm-bindgen introduces newer JS syntax, esbuild catches it automatically via `--target=safari12`
- **Rollback**: Remove the esbuild step from workflows and build script — back to current state

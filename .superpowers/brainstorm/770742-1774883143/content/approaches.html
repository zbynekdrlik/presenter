Stage View Redesign — Approaches

How to fix layout stability and move to WASM

A

WASM Stage with CSS Grid + Container Queries

Full WASM rewrite. Leptos components with CSS Grid for fixed boxes. Rust-side text fitting algorithm. Keep the WYSIWYG editor but make it work properly.

Pros

  • Boxes truly fixed — CSS Grid cells don't shift
  • Text fitting in Rust is type-safe and testable
  • Shared codebase with operator/tablet
  • Remove ~500 lines of embedded JS

Cons

  • Keeps the complex WYSIWYG editor code
  • More configuration surface to maintain
B

Fix Current Server-Rendered HTML + JS

Keep existing approach, fix the text scaling and overflow with better CSS and JS.

Pros

  • Least work — patch CSS/JS without rewriting
  • No WASM dependency for stage
  • Fastest initial page load

Cons

  • Keeps ~500 lines of JS in Rust strings
  • Two rendering stacks (WASM + HTML/JS)
  • Patches symptoms, not root cause
C

WASM Stage with Predefined Layout Presets (Recommended)

Full WASM rewrite + delete the WYSIWYG editor. Replace with 4-6 hardcoded layout presets as Leptos components. Operator picks a preset from a dropdown. No dragging boxes, no appearance sliders.

Pros

  • Everything from A, plus massively simpler code
  • Delete design editor UI, StageDesign/StageBox models, StageAppearance settings
  • Layouts tested as Rust code, not user-configured JSON
  • ProPresenter-like: layouts just work, zero tuning

Cons

  • Less flexible — can't make arbitrary layouts
  • New layouts = Rust code (but that means tested & typed)

Recommendation: C

The WYSIWYG editor was a workaround. You want ProPresenter-like stability. Runtime flexibility is rarely needed. Approach C gives rock-solid layouts with zero configuration, removes complex code, and you get all WASM benefits.

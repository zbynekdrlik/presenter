Stage View Redesign — Approaches

How to fix layout stability and move to WASM

A

WASM Stage with CSS Grid + Container Queries

Recommended. Rewrite stage display as a Leptos WASM component using CSS Grid for fixed-position boxes and a Rust-side text fitting algorithm. Each box is an absolute grid cell that never moves. Text scales via binary search in Rust/WASM (measuring DOM, shrinking until it fits). No JavaScript text scaler.

Pros

  • Boxes are truly fixed — CSS Grid cells don't shift
  • Text fitting in Rust is type-safe and testable
  • Shared codebase with operator/tablet (same Leptos stack)
  • Reactive updates via WebSocket signals — no manual DOM manipulation
  • Remove ~500 lines of embedded JS from server code

Cons

  • Need Safari 12 MVP WASM build (already solved for tablet)
  • Initial WASM load slightly slower than server-rendered HTML
B

Fix Current Server-Rendered HTML + JS

Keep the existing server-rendered approach but fix the text scaling and box overflow. Replace the JS binary-search scaler with CSS overflow: hidden on all boxes + improved font-size: clamp() calculations. Add position: absolute to all stage boxes so they never affect each other's layout.

Pros

  • Least work — fix CSS/JS without rewriting
  • No WASM dependency for stage displays
  • Fastest initial page load

Cons

  • Keeps ~500 lines of JS embedded in Rust strings (maintenance nightmare)
  • CSS clamp() alone can't guarantee text always fits — still needs JS measurement
  • Two rendering stacks (WASM for operator/tablet, HTML/JS for stage)
  • Doesn't solve the root cause — just patches symptoms
C

WASM Stage with Predefined Layout Presets

Like approach A (full WASM rewrite) but remove the WYSIWYG editor entirely and replace with 4-6 hardcoded layout presets. Each preset is a Leptos component with fixed proportions. Operator picks a preset from a dropdown — no dragging boxes. Runtime: just switch preset or toggle an element.

Pros

  • Everything from A, plus massively simpler code
  • Delete the entire design editor UI + StageDesign/StageBox models
  • Delete StageAppearance settings — presets handle everything
  • Layouts are tested as Rust code, not as user-configured JSON
  • ProPresenter-like: layouts just work, no tuning needed

Cons

  • Less flexible — can't make arbitrary layouts
  • Adding a new layout means writing Rust code, not dragging boxes
  • If an artist needs something unusual, you're limited to presets

My Recommendation: C

You said the WYSIWYG editor was a workaround. You want ProPresenter-like stability. You said runtime flexibility is rarely needed. Approach C gives you rock-solid layouts with zero configuration, removes a lot of complex code (design editor, appearance settings, box models), and you get the WASM benefits. If you ever need a new layout, it's a Rust component — tested, typed, reliable.
